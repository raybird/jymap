## Executive Summary

**jymap** 是一個互動式金庸世界歷史地圖平台，整合時間軸、地圖視覺化與事件卡片，讓使用者能夠系統性地探索金庸15部武俠小說與真實歷史事件的對應關係。透過即時同步的時間軸與地圖互動，使用者可以深入理解金庸世界觀的時間與空間脈絡，並為未來的遊戲改編與教育應用奠定基礎。

**核心價值：** 將被動的資料查詢轉變為主動的探索體驗，讓金庸小說與歷史事件更生動地結合，提供更深層的理解與欣賞。

## Core Vision

### Problem Statement

**核心問題：**
金庸小說讀者、歷史愛好者與教育工作者在理解金庸作品時面臨以下挑戰：

1. **地理理解困難：** 閱讀小說時難以直觀理解故事發生的地理位置，無法建立空間概念
2. **歷史對應模糊：** 不清楚小說情節與真實歷史事件的對應關係，缺乏系統性的時空脈絡
3. **探索體驗被動：** 現有解決方案（維基百科、研究資料）都是被動查詢，缺乏互動式探索體驗
4. **資料分散：** 時間線、地理資訊、歷史事件分散在不同來源，難以整合理解

### Problem Impact

**問題影響：**
- **降低閱讀體驗：** 缺乏視覺化支援，降低沉浸感與理解深度
- **限制世界觀認知：** 難以建立完整的金庸世界觀時間與空間認知
- **學習效率低下：** 教育工作者缺乏有效的互動式教學工具
- **探索動機不足：** 被動查詢方式無法激發主動探索的興趣

### Why Existing Solutions Fall Short

**現有解決方案的不足：**

1. **維基百科：** 
   - 資訊完整但缺乏視覺化
   - 文字為主，無法直觀理解時空關係
   - 缺乏互動式探索體驗

2. **研究論文/書籍：**
   - 深度但不易取得
   - 缺乏整合的時間軸與地圖視覺化
   - 無法即時對應歷史事件與小說情節

3. **手動查詢：**
   - 耗時且不系統
   - 需要跨多個平台查詢
   - 難以建立完整的認知框架

**核心差距：** 現有方案都是「被動查詢工具」，缺乏「主動探索平台」的體驗。

### Proposed Solution

**解決方案：**
建立一個整合性的互動式金庸世界歷史地圖平台，包含：

1. **時間軸視覺化：** 從春秋（前473年）到清朝乾隆年間（18世紀）的完整時間線，可拖曳探索
2. **地圖視覺化：** 歷史地圖底圖（Old Maps Online 風格）+ 事件標記點
3. **即時同步：** 時間軸與地圖的即時對應，拖曳時間軸時地圖自動更新標記
4. **事件卡片：** 點擊地圖標記顯示詳細資訊（歷史事件、小說情節、相關人物、資料來源）
5. **資料整合：** 整合 Google Sheets 編年史、維基百科、遠流官方網站等多個權威資料來源

**成功樣貌：**
- 使用者可以透過拖曳時間軸，即時看到地圖上對應的事件標記
- 點擊地圖標記，查看詳細的事件卡片（歷史事件、小說情節、相關人物）
- 系統性地探索整個金庸世界觀的時間與空間脈絡
- 建立對金庸作品的深層理解，為未來遊戲改編與教育應用奠定基礎

### Key Differentiators

**核心差異化優勢：**

1. **整合性體驗：** 時間軸 + 地圖 + 事件卡片的完整整合，而非分散的工具
2. **互動式探索：** 從「被動查詢」轉變為「主動探索」的體驗模式
3. **即時同步：** 時間軸與地圖的即時對應，提供流暢的視覺化體驗
4. **資料完整性：** 整合多個權威資料來源（維基百科、Google Sheets 編年史、遠流官方網站）
5. **視覺化設計：** 參考 Old Maps Online 的歷史地圖風格，提供沉浸式體驗
6. **擴展性：** 未來可改編成遊戲、教育工具、社群平台等

**競爭優勢：**
- **技術優勢：** 現代地圖視覺化技術已成熟，可實現流暢的互動體驗
- **資料優勢：** 整合多個權威資料來源，確保資料完整性與準確性
- **體驗優勢：** 獨特的「探索模式」而非「查詢模式」，提供差異化價值

### Market Analysis (from Party Mode Discussion)

**目標使用者群體：**

1. **金庸小說讀者（核心群體）**
   - 特徵：已讀過部分或全部作品，想深入理解世界觀
   - 痛點：閱讀時難以對應地理位置與歷史事件
   - 需求：視覺化探索、時空脈絡理解
   - 規模：潛在用戶基數大（華語圈數千萬讀者）

2. **歷史愛好者（次要群體）**
   - 特徵：對中國歷史有興趣，想了解小說與史實的對應
   - 痛點：缺乏整合的視覺化工具
   - 需求：歷史事件與小說情節的對應關係
   - 規模：中等，但參與度高

3. **教育工作者（潛在群體）**
   - 特徵：需要教學工具，幫助學生理解文學與歷史
   - 痛點：缺乏互動式教學資源
   - 需求：可作為教學輔助工具
   - 規模：較小但付費意願可能較高

**市場規模估算：**
- 目標市場：對互動式探索有興趣的讀者，估計 10-50 萬
- 主要使用者：金庸讀者（核心）+ 歷史愛好者（次要）+ 教育工作者（潛在）

**市場定位：**
- **主要定位：** 金庸讀者的「探索工具」
- **次要定位：** 歷史愛好者的「學習平台」
- **潛在定位：** 教育市場的「教學工具」

**競爭優勢：**
- 整合性：時間軸 + 地圖 + 事件卡片的完整整合
- 互動性：即時同步的視覺化體驗
- 資料完整性：多源整合的權威資料
- 視覺化：歷史地圖風格的沉浸式體驗

### User Journey

#### 典型使用者旅程（以金庸群俠傳Online玩家為例）

**階段 1：發現（Discovery）**
- **觸發點：** 遊戲中遇到「襄陽城」或「襄陽之戰」相關劇情
- **行為：** 想了解真實的襄陽城在哪裡，歷史上的襄陽之戰是什麼
- **痛點：** 需要跨多個平台查詢，資訊分散

**階段 2：首次使用（Onboarding）**
- **行為：** 進入 jymap，看到時間軸與地圖
- **體驗：** 拖曳時間軸到「南宋 1230s」，地圖自動顯示襄陽城標記
- **價值：** 即時看到地理位置與時間對應

**階段 3：核心使用（Core Usage）**
- **行為：** 點擊地圖上的「襄陽城」標記
- **體驗：** 彈出事件卡片，顯示：
  - 歷史事件：襄陽保衛戰（1230s）
  - 對應小說：射鵰英雄傳、神鵰俠侶
  - 相關人物：郭靖、黃蓉、蒙古大軍
  - 資料來源：維基百科連結
- **價值：** 完整理解事件的地理、時間、歷史、小說對應關係

**階段 4：深入探索（Deep Exploration）**
- **行為：** 透過時間軸探索相關事件（如「蒙古入侵」、「郭靖楊康結義」）
- **體驗：** 建立完整的時空脈絡認知
- **價值：** 系統性理解整個金庸世界觀

**階段 5：應用與分享（Application & Sharing）**
- **行為：** 將理解應用到遊戲體驗中，或與其他玩家分享
- **體驗：** 提升遊戲沉浸感，建立社群討論
- **價值：** 從「玩遊戲」提升到「理解世界觀」的層次

#### 關鍵時刻（Aha Moments）

1. **地理對應：** 「原來襄陽城在這裡！難怪是戰略要地」
2. **時空連結：** 「原來射鵰和神鵰是連續的故事，都在南宋時期」
3. **歷史對應：** 「原來小說中的事件對應真實的歷史戰役」
4. **世界觀建立：** 「現在我理解整個金庸世界的時空脈絡了」

## Success Metrics

### User Success Metrics

**核心使用者成功指標：**

1. **時空脈絡認知建立**
   - **指標：** 使用者透過時間軸探索，建立完整的時空脈絡認知
   - **測量方式：** 
     - 使用者單次會話中探索的事件數量（目標：平均 3+ 個事件）
     - 使用者跨不同朝代的探索行為（目標：60% 使用者探索 2+ 個朝代）
     - 使用者從單一事件擴展到相關事件的探索行為（目標：40% 使用者會點擊相關事件連結）

2. **多事件探索行為**
   - **指標：** 使用者會探索多個事件，而非單一查詢
   - **測量方式：**
     - 平均每次會話探索的事件數（目標：3-5 個事件）
     - 使用者使用時間軸拖曳的頻率（目標：80% 使用者會拖曳時間軸）
     - 使用者點擊多個地圖標記的比例（目標：70% 使用者點擊 2+ 個標記）

3. **核心探索旅程完成**
   - **指標：** 使用者完成核心探索旅程的比例
   - **測量方式：**
     - 完成完整探索旅程的使用者比例（目標：50% 使用者完成完整旅程）
     - 核心旅程定義：發現 → 首次使用 → 核心使用 → 深入探索
     - 使用者從「查詢單一事件」到「系統性探索」的轉換率（目標：30% 轉換率）

4. **關鍵時刻達成**
   - **指標：** 使用者達到「Aha Moments」的比例
   - **測量方式：**
     - 使用者達到「地理對應」時刻的比例（目標：80%）
     - 使用者達到「時空連結」時刻的比例（目標：60%）
     - 使用者達到「世界觀建立」時刻的比例（目標：40%）

## MVP Scope

### Core Features

**MVP 核心功能（必須包含）：**

1. **時間軸與地圖同步（MVP 核心）**
   - **功能描述：** 時間軸拖曳時，地圖自動更新顯示對應時間範圍的事件標記
   - **技術要求：** 即時同步，無明顯延遲（< 200ms）
   - **使用者價值：** 創造核心「時空連結」體驗，讓使用者理解時間與空間的對應關係

2. **事件標記（地圖顯示）**
   - **功能描述：** 地圖上顯示事件標記，點擊標記顯示事件卡片
   - **技術要求：** 支援大量標記點（100+ 事件），效能優化渲染
   - **使用者價值：** 直觀的地理位置視覺化，對應產品願景「更容易理解地理參考」

3. **事件卡片**
   - **功能描述：** 點擊地圖標記或時間軸事件，顯示詳細事件資訊卡片
   - **內容包含：** 事件名稱、歷史年份、對應小說情節、地理位置、相關連結
   - **使用者價值：** 提供深入資訊，支援「多事件探索」行為

4. **時間軸拖曳功能**
   - **功能描述：** 使用者可以拖曳時間軸，選擇特定時間範圍
   - **技術要求：** 流暢的拖曳體驗，支援快速跳轉到特定朝代
   - **使用者價值：** 核心探索功能，支援「時空脈絡認知建立」

5. **地圖縮放與平移**
   - **功能描述：** 標準地圖互動功能（縮放、平移、拖曳）
   - **技術要求：** 使用 Leaflet 或類似地圖庫，支援多層級縮放
   - **使用者價值：** 基本地圖操作，支援地理探索

6. **搜尋功能**
   - **功能描述：** 使用者可以搜尋事件、小說、人物、地點
   - **技術要求：** 即時搜尋建議，支援模糊搜尋
   - **使用者價值：** 快速定位特定內容，提升探索效率

7. **資料完整性（15 部作品）**
   - **功能描述：** MVP 必須涵蓋所有 15 部金庸作品的事件資料
   - **資料來源：** 以 Wikipedia 調研資料為主，整合 Google Sheets 編年史
   - **資料要求：** 
     - 所有事件具備歷史年份
     - 所有事件具備地理座標（盡可能精確）
     - 所有事件具備對應小說情節說明
   - **使用者價值：** 完整的金庸世界觀，支援「跨朝代探索」

8. **視覺風格（Old Maps Online 風格）**
   - **功能描述：** 地圖視覺呈現參考 Old Maps Online 的復古地圖風格
   - **技術要求：** 使用復古地圖圖層或自訂地圖樣式
   - **使用者價值：** 創造沉浸式歷史體驗，強化「歷史事件對應」感受

### Out of Scope for MVP

**明確排除的功能（v2.0 或之後）：**

1. **使用者帳號與個人化**
   - 使用者登入/註冊功能
   - 個人收藏/書籤功能
   - 使用歷史記錄
   - **理由：** MVP 專注於核心探索體驗，個人化功能可延後

2. **社群功能**
   - 使用者評論/討論
   - 分享功能（社交媒體整合）
   - 使用者生成內容（UGC）
   - **理由：** 先驗證核心價值，再考慮社群擴展

3. **進階分析功能**
   - 事件關聯分析
   - 人物關係圖
   - 時間線比較工具
   - **理由：** 屬於進階功能，超出 MVP 範圍

4. **多語言支援**
   - 英文介面
   - 其他語言版本
   - **理由：** MVP 專注於中文使用者，多語言可延後

5. **行動應用程式（App）**
   - iOS/Android 原生應用
   - **理由：** MVP 先以 Web 應用為主，行動應用可延後

6. **離線功能**
   - 離線資料存取
   - PWA 離線支援
   - **理由：** MVP 專注於線上體驗，離線功能可延後

7. **進階視覺化**
   - 3D 地圖
   - 動畫效果
   - 虛擬實境（VR）體驗
   - **理由：** 超出 MVP 範圍，屬於未來願景

8. **API 與資料匯出**
   - 開放 API
   - 資料匯出功能（JSON/CSV）
   - **理由：** MVP 專注於使用者體驗，API 可延後

   - ✅ 響應式設計（桌面與平板裝置）
   - ✅ 資料載入與渲染效能符合標準

**MVP 決策點：**

- **Go/No-Go 標準：** 如果達到以上 80% 的成功標準，則可進入下一階段（成長階段）
- **迭代標準：** 如果未達到標準，則需要針對特定功能進行迭代優化
- **擴展標準：** 如果達到所有標準且使用者反饋積極，則可開始規劃 v2.0 功能
